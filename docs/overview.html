Statistically indistinguishable

The core duality

Two capabilities, equal weight, one engine.

Both read the same Profile and the same declarative config. One makes data you can trust; the other makes the data that breaks systems in production — so you catch it first.

Faithful Generator

Real, minus the risk

Reproduces distributions, correlations, and referential integrity — statistically identical, provably free of real values.

  • Marginals + in-house Gaussian copula for correlations
  • Multi-table FK integrity, realistic Faker values
  • Leakage gate: exact-membership check, fail-closed
  • Conditional & seeded — byte-identical per seed
Data Chaos Monkey

Broken on purpose

Systematically injects out-of-distribution, malformed, and schema-breaking data under a declarative policy — with a full audit of every fault.

  • Out-of-distribution & format / type violations
  • Schema & constraint breakage (orphan FKs, dup keys…)
  • Mixed-rate or fully-chaotic policy
  • Bidirectional fault-manifest audit
End to end

Connect once. Everything downstream reads a Profile.

The Profile is a statistical description — never raw values — so you produce it once and generate, faithful or chaotic, offline and anywhere. No real data ever leaves the source.

Faithful Generator

Statistically real, not just plausible.

Distributions

Marginal synthesis

Per-column reproduction from the Profile: histogram inverse-transform, category frequencies, datetime range, length-faithful text. Nulls preserved.

Correlation

In-house Gaussian copula

Numeric cross-column correlations (Spearman) preserved with a hand-rolled copula — no SDV/Copulas (BUSL license excluded).

Integrity

Referential integrity

generate_related builds tables parents-before-children with unique PKs and valid FKs, incl. junction & self-referential.

Control

Conditional & seeded

--where conditions a column (equality, range, set); 100% of rows satisfy it while others keep their shape. Same seed → byte-identical.

Privacy

Leakage gate

Sensitive columns hashed (keyed BLAKE2b) into a guard; every generated value is checked against the real set, regenerated on collision, fail-closed otherwise.

Realism

Realistic values

Text that looks like email / name / phone / id gets coherent Faker values, so the output reads like real records — not noise.

Data Chaos Monkey · the differentiator

The malformed data your test suite never covers.

Every fault is deterministic, parameterised, and recorded in a Fault Manifest — so a broken row is a designed test case, not a surprise.

Engine

Pluggable mutators

Mutators are discovered from the tymi.mutators entry-point group and run as an ordered chain, each merging its manifest. A new fault type needs zero core changes.

Out-of-distribution

Outlier injection

Pushes a configurable proportion of numeric/datetime cells out of range (max + magnitude·span), integer-safe and null-preserving.

Format & type

Value violations

Five toggleable faults: text_in_numeric, invalid_date, broken_encoding, oversized_string, illegal_null.

Schema & constraint

Structural breakage

Six structural faults: missing / extra / renamed field, changed_type, duplicate_keys (composite-aware), orphan_fk.

Policy

Declarative chaos

mixed corrupts a rate fraction of rows and leaves the rest faithful; fully_chaotic breaks the whole table (and demands --confirm over FKs).

Audit

Bidirectional manifest

--audit verifies both directions: every listed fault materialised, and every change vs the faithful baseline is listed. A manifest that lies fails the build.

Privacy & evaluation

Prove it's faithful — and prove nothing leaked.

Detection

PII auto-classification

Rules-based detection (email / SSN / IBAN / credit-card [Luhn] / IP / phone + column-name hints) feeds the leakage guard automatically.

Filters

Similarity & outlier

Drop-only privacy filters remove any generated row too close to a real one, or clustered memorised extremes — deterministic, schema-preserving.

Fidelity

Fidelity report

Per-column KSComplement / TVComplement + global correlation similarity, all in-house; --tolerance gates a CI build and names the failing columns.

Risk

Quality & privacy report

A composite quality score plus membership-disclosure and attribute-inference metrics — three configurable CI gates that fail the build on breach.

Whole-DB provisioning · PRD 1

A whole obfuscated database, in one command.

The MVP works per table. This lifts it to the whole database — with cross-team consistency, pinned login fixtures, and a fail-closed guardrail — behind tymi provision --spec, runnable in CI or an Airflow DAG.

1

Load the Spec AD-14

A versioned artifact bundling every table's pinned Profile, seed, tolerance, shared-key decls, reserved keyspace, and fixtures — read offline, never re-profiling the source.

2

Destination guardrail — fail closed AD-18

Requires an explicit environment: nonprod affirmation and no prod-deny-list match. Missing affirmation aborts; empty deny-list never means “allow all”.

3

Whole-DB faithful generation AD-13

Every table in FK-topological order, each with the leakage gate embedded — zero real sensitive values.

4

Per-table RNG substreams AD-20

Each table draws from a stream keyed by (seed, table_name) — rows and FK edges stable no matter what unrelated tables do.

5

Shared-key emission AD-16

Columns declared shared become position-derived keys — independent of source and seed — so two teams get identical join keys.

6

Fixtures overlay + scan-and-reject AD-17

Pinned login rows injected verbatim in the reserved keyspace — never regenerated, but scanned for real values & PII. A hit fails closed.

7

Seal into a GatedDataset AD-21

A type mintable only by the gate. The load boundary accepts a GatedDataset and refuses a raw one — un-gated data at a destination is a type error.

8

Idempotent load → report AD-15

Each table clean-replaced (a re-run self-heals), then a report: rows, fixtures, gated columns, fidelity — and the consistency-unit fingerprint.

The wedge

Two teams, one synthetic reality.

Reproducible identity is a consistency unit: the Spec, its pinned Profiles, the seed, the pinned dependency versions, and the fixture set. Share the unit and you get byte-identical data — and a fingerprint that only matches when the reality truly matches. A drifted source or bumped dependency shows up as a different fingerprint instead of silently mismatched entities.

The artifact

One editable YAML describes the whole database.

Bootstrapped once from a source, then hand-edited. The Profiles are embedded, so regeneration is fully offline.

schema_version: "1.0.0"
seed: 42
tolerance: 0.9
destination:
  environment: nonprod      # the affirmation
  host: dev-db.internal
  database: app_dev
tables:
  customers:
    profile: { ... pinned, offline ... }
    rows: 10000
    shared_keys: [customer_id]   # cross-team
    reserved_key_block: 1000    # fixtures live below
    fixtures:
      - { customer_id: 1, email: "qa.admin@test.dev" }

What each field buys you

profile — the statistical mold, embedded so no source is touched again.

shared_keys — the columns that must line up across teams.

reserved_key_block — the low key range fixtures own; generated keys start above it, disjoint by construction.

fixtures — the exact accounts people need to log in, pinned verbatim and still scanned.

destination — the single owner of the non-prod guardrail; excluded from the fingerprint.

Phase 2 & 3 · scale & controls

From a few MB to hundreds of TB — and finer control.

The Phase-1 seam pays off: because keys are position-derived, a child resolves its foreign keys from a parent's position without ever loading the parent — so the same engine streams a database of any size, and knows what's unchanged on a re-run.

Out-of-core · AD-22–24

Streaming, bounded memory

tymi provision --stream generates and writes the whole DB chunk-by-chunk — peak memory is one row-chunk regardless of total size. First chunk replaces, the rest append (idempotent).

Position-addressable

FKs without the parent

A child block draws parent positions and maps them to keys by formula (surrogate pos, shared reserved+pos) — referential integrity holds with the parent never in RAM.

Correlation · AD-25

Cross-table dependency

A declared amount ↔ customer.tier is induced by a rank reorder that hits Spearman ≈ ρ against the referenced parent value — while preserving the child's marginal.

Subset · AD-26

A smaller, valid DB

Keep a fraction of a root table; descendants connected to it and the dimensions they need come along, RI intact. Keys aren't renumbered, so a subset still joins to a full dataset.

Delta · AD-27

Refresh only what changed

Diff two Specs and regenerate only the dirty tables — a key-affecting parent dirties its children one hop, a correlation parent dirties its correlated child. A clean table is byte-identical to before.

Honest identity

The fingerprint tells the truth

In-memory and streaming emit different byte layouts, so the consistency-unit fingerprint folds in the mode — it never claims two different realities are the same.
