TYMI PRD-1 Phase-1

Obfuscated prod-like dev environments — whole-DB, self-service, zero real values. Inherits the MVP spine (AD-1…12, hexagonal) read-only; adds AD-13…21.

The bet, in one line

The differentiated value is not whole-DB faithful generation (SDV/Tonic do that) — it is cross-team consistency from one shared, versioned spec with zero real values. Phase 1 exists to validate that bet cheaply (a few tables, two teams, the current in-memory engine) before funding the out-of-core rewrite (Phase 2) or cross-table correlation (Phase 3).

Phasing

Phase 1 — validate the wedge
Whole-DB FK-consistent generation at small scale, leakage gate, fixtures, cross-team consistency, safe self-service provision. PDE-1..5, 7..15.
Phase 2 — scale (rewrite)
Out-of-core / chunked, table-parallel generation; leakage set → Bloom/disk; BLAS pinning. NFR-A/B. A rewrite, not an extension.
Phase 3 — depth & controls
Cross-table correlation (PDE-6), subsetting (PDE-16), delta refresh (PDE-17). Each its own project.

The new architecture decisions

ADDecisionPrevents (the divergence)
AD-13Whole-DB gen first-wires the existing generate_related (topological, FK-aware)a second, divergent multi-table generator
AD-14The Spec = one versioned superset of Config (Profiles + fixtures + keys + dest + seed)divergent spec formats; where fixtures/sensitive live
AD-15Identity = consistency unit (Spec + pinned Profile artifacts + seed + deps); regenerate offline, never re-profile"same spec, drifted source → garbage joins"; a lying fingerprint
AD-16Shared keys are position-derived, source- and seed-independent; fixtures in a declared disjoint reserved block, validatedcross-team key drift; fixture/key collision
AD-17Fixtures: inject-verbatim, regenerate-never, scanned (guard + PIIClassifier, fail closed) into the GatedDatasetfixtures as a PII bypass around the gate
AD-18Provision fails closed unless the Spec affirms a non-prod destination (+ prod deny-list); obfuscation never optionalwriting to prod; a soft "obfuscate?" checkbox; two guardrail owners
AD-19provision is a thin composition-adapter pipeline; the CLI & a DAG call it identicallyorchestration creep into core; two flow implementations
AD-20Per-table RNG substreams from (seed, table) — a table's output is independent of other tables' row counts/orderan unrelated table shifting shared RNG → same keys, different relationships → garbage joins
AD-21The load boundary accepts only a GatedDataset — un-gated data at a destination is a type errorpost-gen overlays (keys/fixtures) reaching the destination un-scanned

What we build (Phase 1)

  • tymi/config/spec.py — the whole-DB Spec model + auto-bootstrap (bundles Profiles)
  • tymi/provision/pipeline.py + guardrail.py — the composition-adapter pipeline + destination guardrail
  • tymi/synth/substreams.py, keys.py, fixtures.py — per-table RNG, shared keys, fixtures overlay
  • tymi/synth/leakage.py — a scan-and-reject gate mode; tymi/domain/artifacts.py — the GatedDataset type
  • tymi/cli/app.py — the provision command (thin call)

No new runtime dependency; generation/introspection/load reuse shipped code. generate_related exists but has no callers today — this is first-wiring.
